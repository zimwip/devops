01 — Architecture

Overview

Each microservice is an autonomous unit with its own lifecycle: its own Git repo, Jenkins pipeline, semantic version, and Helm chart. The platform enforces global consistency through shared conventions and a central environment configuration repository.

Key principle

Each service deploys independently. A production bug in service-auth does not require redelivering service-orders.

Source of truth

The platform-config repo declares which version of each service runs in which environment.

Hybrid portability

Helm abstracts the target. Only values-{env}.yaml differs between OpenShift and EKS.

Shared library

Common CI/CD logic lives in jenkins-shared-lib, not duplicated in every Jenkinsfile.

04 — Traceability

Environment Management

A versioned platform-config Git repo is the single source of truth for all environments. Every change goes through a PR — giving a complete, auditable history of every deployment.

versions.yaml format

# prod/versions.yaml
_meta:
  updated_at: "2026-03-28T14:32:00Z"
  updated_by: "jenkins/job/platform-deploy/42"
  env_type:   "fixed"
  cluster:    "openshift-prod"

services:
  service-auth:
    version:     "2.2.1"
    image:       "registry.internal/service-auth:2.2.1"
    deployed_at: "2026-03-28T14:32:00Z"

Full traceability via git log prod/versions.yaml — every commit corresponds to a deployment, with the approver recorded via the GitHub PR.

Cluster profiles

The platform.yaml at the root of the platform repo defines all known clusters. Each profile drives the full deploy path — which CLI tool to use, which registry to pull from, and which Helm values file to apply.

# platform.yaml — cluster profiles
clusters:
  openshift-dev:
    platform:           "openshift"   # "openshift" | "aws"
    api_url:            "https://api.ocp-dev.internal:6443"
    context:            "ocp-dev/system:admin"
    registry:           "registry.internal"
    helm_values_suffix: "dev"   # → helm/values-dev.yaml

  eks-prod:
    platform:           "aws"
    region:             "eu-west-1"
    cluster_name:       "platform-eks-prod"
    registry:           "123456789.dkr.ecr.eu-west-1.amazonaws.com"
    helm_values_suffix: "prod"

OpenShift deploy path

Switches kubeconfig context via oc config use-context, then runs helm upgrade --install with --set openshift.enabled=true. Requires an active oc login session or a pre-configured kubeconfig context.

AWS / EKS deploy path

Updates kubeconfig automatically via aws eks update-kubeconfig, then runs helm upgrade --install with --set aws.region=.... Image pull credentials come from IRSA or the node instance profile — no explicit docker login needed.

Platform resolution order

Explicit --platform flag → cluster profile in platform.yaml → base environment's platform → fallback to openshift. The resolved platform is stored in versions.yaml so every subsequent deploy uses the same target automatically.

Jenkins integration

When JENKINS_TOKEN is set, Jenkins receives PLATFORM and CLUSTER as build parameters and handles the platform-specific auth internally. The local deploy path is used only for POC environments and local testing.

05 — Ephemeral Environments

POC Environments

Any developer can spin up an isolated namespace forked from a base environment, deploy experimental service versions, iterate freely, then destroy everything cleanly. The entire lifecycle is tracked in Git.

POC versions.yaml anatomy

# envs/poc-payment-kafka-20260403/versions.yaml
_meta:
  env_type:          "poc"         # distinguishes from fixed envs
  base_env:          "staging"     # forked from here
  owner:             "john.doe"
  description:       "Test async payment flow with Kafka"
  expires_at:        "2026-04-17T00:00:00Z"    # soft deadline (default 14 days, max 365)
  branch_convention: "poc/payment-kafka"
  cluster:           "openshift-dev"
  namespace:         "platform-poc-payment-kafka-20260403"

services:
  service-auth:     # inherited from staging — unchanged
    version: "2.3.0"
  service-payments: # experimental — only exists here
    version:      "0.1.0-SNAPSHOT-exp42"
    experimental: true

Complete CLI workflow

# 1. Create a POC forked from staging
python scripts/platform_cli.py env create \
  --name payment-kafka --type poc --base staging \
  --description "Test async Kafka payment flow" --ttl-days 14

# Windows CMD
platform.bat poc create payment-kafka staging

# Windows PowerShell
.\platform.ps1 poc create payment-kafka staging

# 2. Deploy an experimental version to your POC
python scripts/platform_cli.py deploy \
  --env poc-payment-kafka-20260403 \
  --service service-payments --version 0.1.0-SNAPSHOT-exp42

# 3. Diff your POC against staging
python scripts/platform_cli.py env diff \
  --from poc-payment-kafka-20260403 --to staging

# 4. Destroy when done
python scripts/platform_cli.py env destroy \
  --name poc-payment-kafka-20260403 --force

Git = audit log

Every POC creation and destruction is a Git commit in platform-config. Who created it, when, and what versions were deployed — all traceable.

Quality gate skipped

poc/* branches skip SonarQube in Jenkins. Fast iteration without coverage blockers.

Soft TTL deadline

Expiry is a soft deadline — no automatic destruction. When a POC passes its expires_at, a warning appears in env info and the dashboard. Extend with env extend or destroy manually.

Promote to staging

When the experiment is validated, merge poc/payment-kafka into develop, then create a release/* branch — the normal promotion flow takes over.

06 — Compatibility

Inter-Service Dependencies

Each service explicitly declares the minimum required versions of its dependencies in a service-manifest.yaml. Jenkins validates all constraints before every deployment.

service-manifest.yaml

# service-orders/service-manifest.yaml
name: service-orders
version: "1.9.0"

requires:
  service-auth: ">=2.2.0"
  service-notif: ">=3.0.0"
  lib-platform: ">=1.3.0"

exposes:
  api-version: "v2"
  contract: "openapi/orders-v2.yaml"

08 — Changelog

Release Notes Generation

Release notes are generated automatically from commit messages using Conventional Commits. Each service publishes its own GitHub Release, independently from the others.

Conventional Commits convention

feat(orders): add deferred payment support
fix(orders): correct VAT calculation on composite products
chore: upgrade Spring Boot to 3.4
BREAKING CHANGE: /orders API now returns a paginated object

# Format: type(scope): description
# Types: feat | fix | chore | docs | refactor | test | perf

CLI access

# Fetch the latest GitHub Release for a service
python scripts/platform_cli.py release-notes --service service-auth
python scripts/platform_cli.py release-notes --service service-auth --version 2.2.1 --json

Result

One GitHub Release per service per version, with the diff since the previous release, the Docker image, and the Git SHA.

Prerequisite

Enforce Conventional Commits via a pre-commit hook (commitlint) or a GitHub Action validating PR titles.

Platform CLI

python scripts/platform_cli.py release-notes --service my-svc fetches release notes directly from the terminal on all platforms.

09 — Platform

Cross-cutting Concerns

These capabilities are never re-implemented in each service. They are either injected by the platform (transparent to the service) or provided via the internal lib-platform Maven library.

Logging

The service writes to stdout in JSON. The Fluentd/Vector sidecar collects, enriches, and forwards. Zero logging config in the service.

Security

mTLS between services managed by Istio. User authentication (JWT/OIDC) via a Spring Security filter in lib-platform. Services never touch Keycloak directly.

Secrets

Vault or AWS Secrets Manager injects secrets as environment variables at pod startup. The service reads System.getenv("DB_PASSWORD"). No secrets in Git — ever.

Tracing

The OpenTelemetry agent attaches to the JVM via the Dockerfile. It auto-instruments Spring Boot, Kafka, and JDBC. The service writes zero tracing code.

Configuration

ConfigMaps for infrastructure config. Spring Cloud Config Server for application config with hot-reload. Property names follow a shared naming convention.

Pub/Sub

Topic naming: orders.order.created. Schema Registry versions Avro schemas. lib-platform provides producers/consumers with retry and dead-letter queue.

lib-platform — the platform glue

<!-- pom.xml of every Java service -->
<dependency>
  <groupId>com.myorg</groupId>
  <artifactId>lib-platform</artifactId>
  <version>${lib-platform.version}</version>
</dependency>

# What lib-platform activates automatically:
+ JWT/OIDC security filter (Keycloak)
+ Kafka producer/consumer configuration + DLQ
+ JSON structured log format (MDC: traceId, serviceVersion)
+ /actuator/info enriched (version, commit, env)
+ OpenTelemetry propagation (W3C TraceContext)
+ Unified HTTP error handling (RFC 7807)
